# Editorial Brief — Standing Rewrite Prompt

This is the standing brief for rewriting any section of `site-copy.md`. It is not a one-off prompt — apply it every time copy gets touched, so the whole site reads like it was written by the same person.

This is **not** a copyediting task. It is an editorial rewrite.

The job is to preserve Aria's ideas while making the writing sound unmistakably human, intellectually curious, technically credible, and memorable.

---

The audience includes:

The writing should work for all of them.

## The biggest problem with the old copy

It is too abstract.

It hides concrete ideas behind technical nouns.

It often sounds like someone trying to sound intelligent instead of someone genuinely explaining what fascinated them.

Rewrite everything to feel like a real person speaking.

## Core writing principles

Every section should answer:

- Why did I build this?
- What bothered me?
- What question wouldn't leave me alone?
- What changed because I built it?

Don't start with features. Start with curiosity. Then explain the artifact. Then explain what was learned.

## Keep

Her personality. Curiosity. Cross-disciplinary thinking. History. Language. Psychology. Design. Roman logistics. Museums. Memory.

These are features, not distractions. They explain how she thinks.

---

## Remove

Overly abstract phrases. Empty sophistication. Needlessly poetic sentences. Words that sound impressive but communicate little.

Examples: "substrate," "coordination," "infrastructure," "verification," "artifact" — unless immediately grounded in something concrete.

(Grounded means: the word appears right next to a specific, checkable detail — a project name, a number, a mechanism — not floating alone as a vibe.)

## Rewrite philosophy

Don't polish. Rebuild.

If a paragraph can be improved by deleting 80% of it and rewriting from scratch, do that. Keep only ideas worth keeping.

**Editorial license: don't treat the existing copy as sacred.** If an entire section should disappear, merge, or get reorganized, say so and do it. Disagreement with the current draft is usually where the best editing happens — flag it, argue for it, don't just smooth the surface.

## Sentence-level guidance

Prefer:

- "I kept noticing..."
- "I became obsessed with..."
- "It bothered me that..."
- "I wanted..."
- "I eventually realized..."
- "I built..."
- "I learned..."

Avoid (unless genuinely necessary):

- "This project leverages..."
- "This solution enables..."
- "The goal is to..."
- "This platform provides..."
- "The architecture consists of..."

## Compression Pass (Critical)

This is a second-pass brief. Run it after the main rewrite above has already happened — on a fresh pass, not blended into the first one.

The first pass gets the ideas onto the page in the right voice. This pass is not another rewrite. It is an aggressive edit: imagine cutting a magazine feature down by 40% while making it *more* memorable.

### The rule

If a paragraph can lose 30-50% of its words without losing meaning, cut it.

Don't be afraid of white space. Don't be afraid of short paragraphs. Don't be afraid of stopping early. Readers are smart. Trust them.

### Think Apple, not documentation

Think Stripe. Think Paul Graham. Think excellent magazine writing.

Not feature lists. Not README files. Not product brochures.

The writing should feel confident enough to leave things unsaid.

### Compress every section

Most project descriptions currently follow: problem → solution → implementation → evidence. That's too long.

Instead aim for: one memorable observation → one paragraph explaining what was built → one sentence explaining why it matters. That's usually enough.

### Kill repetition

Every project can't open with "I noticed...", "I kept seeing...", "I got tired of...", "That bothered me..." The pattern becomes obvious fast. Vary the rhythm. Sometimes open with a surprising observation, a contradiction, a bold statement, a single memorable sentence.

Examples:

- Every Claude Code session felt like onboarding a brilliant coworker with amnesia.
- Prompt engineering accidentally became folklore.
- Research papers don't belong in lists.
- AI agents are surprisingly good at individual work. They're surprisingly bad at teamwork.

Those create curiosity immediately.

### Prioritize memorable sentences

Every project should contain one sentence someone could tweet. One sentence someone remembers. One sentence that captures the whole idea.

Examples:

- Every paper deserves an address.
- Prompt tricks expire. Mental models compound.
- Memory should prove itself before it gets promoted.
- The bottleneck isn't intelligence. It's coordination.

If a sentence isn't doing that kind of work, question whether it needs to exist.

### Explain less

Don't explain every feature. Mention enough to establish credibility, then move on. Instead of listing seven implementation details, mention two. Trust readers to infer competence.

### Shorter is usually stronger

Prefer "I remember where ideas live." over "I found myself organizing research spatially rather than alphabetically."

Prefer "Every Claude session started from zero." over "I became frustrated with repeatedly reconstructing conversational context across independent sessions."

### Delete throat-clearing

Cut phrases like "I wanted to...", "The point is...", "The goal was...", "The idea behind...", "This project...", "Built around..." Start with the interesting thing.

### Preserve mystery

Don't answer every question immediately. Let readers become curious. A little tension is good.

### The final test

After editing each project, ask: could this lose another 25%? If yes, keep cutting.

The finished site should feel sharp. Fast. Confident. Readable in one sitting. Memorable weeks later.

The goal is not to explain everything. The goal is to make someone want to keep reading.

### Closings, not labels

Do not use `Evidence:` as a visible heading on project cards. Each project ends with one standalone emphasized closing line, no label.

Examples:

- Every paper deserves an address.
- Prompt tricks expire. Mental models compound.
- Memory should prove itself before it gets promoted.

These are brand assets. Lean into them.

### Rhythm breaks

Not every project should follow hook → paragraph → implementation → closing. After several cards in the same shape, the reader predicts the beat. Break it on purpose: three sentences only, a tiny story, a question as the opener, a bold contradiction with white space after it.

### Personality texture

Compression should not sand away what makes the voice identifiable. Rome, museums, old books, grain logistics beside sqlite, cross-disciplinary obsessions: they should appear occasionally, not dominate. The site should read like Aria, not a very good generic AI engineer.
